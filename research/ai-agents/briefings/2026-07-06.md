# Briefing: AI Agents — 2026-07-06

## Summary

New high-signal findings: 5.

This run found a cluster around agent reliability/governance rather than model capability: static analysis for runaway agent loops, static dependency graphs for agent programs, long-term memory as governed evolving state, a LangGraph checkpoint durability fix, and Codex network-permission attribution for concurrent execs.

## Top findings

1. **Infinite Agentic Loops / IAL-Scan** — A new arXiv paper defines agent-specific non-termination failures and reports 68 confirmed failures across 47 projects from a 6,549-repository scan. This is directly relevant to cron and long-running agent safety.
2. **AgentFlow / Agent Dependency Graphs** — A companion static-analysis direction models agents, prompts, tools, memory, models, and handoffs as a graph for governance, BOM generation, and prompt-to-tool risk detection.
3. **Governed Evolving Memory** — “Is Agent Memory a Database?” argues long-term agent memory correctness is about state trajectory, revision, forgetting, and retrieval rather than record-level storage; this is highly relevant to Markdown/SQLite knowledge loops.
4. **LangGraph 1.2.8** — Fixes a fresh-thread DeltaChannel bug where the first `update_state` / `bulk_update_state` write could be silently dropped.
5. **OpenAI Codex network attribution** — Adds Linux execution-scoped attribution tokens so concurrent managed-network requests can be tied to the exact exec call/command that caused them.

## What changed since last run

- The prior run emphasized background-agent product reliability, configurable multi-agent delegation, persistent-state coding-agent risks, and proxy benchmarks.
- This run adds a stronger static-analysis/governance theme: detect loops before deployment, build dependency graphs for agent programs, and attribute dangerous tool/network effects to exact execution scopes.
- The active memory topic progressed from code-memory retrieval (ContextSniper) to a broader data-management theory of governed evolving memory.
- The coding-agent evaluation thread gained more evidence that long-running agents need repository-/system-level metrics, not only isolated task success.

## Open questions

- Can IAL-style checks be added as a scheduled-agent preflight lint for loop limits, tool budgets, and external side-effect guards?
- Which agent frameworks can export ADG-like dependency graphs directly?
- How should Markdown-first memory loops represent revision and forgetting while preserving an audit trail?
- Should scheduled LangGraph agents verify first-checkpoint persistence after initialization?
- Can execution-scoped attribution generalize beyond network access to file writes, process trees, browser actions, and MCP/tool calls?
